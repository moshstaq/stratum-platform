# ADR-001: Data Sources Over Remote State for Cross-Repository References

**Date:** June 2026  
**Status:** Accepted  
**Author:** Moshood Adisa

## Context

stratum-platform consumes outputs from two independently managed platform foundations: azure-landing-zone and aws-landing-zone. A strategy was required for how this repository references infrastructure provisioned by those foundations — specifically VNet IDs, subnet IDs, resource group names, and IAM role ARNs that workload modules need to deploy into the correct platform boundaries.

Two mechanisms exist for cross-repository Terraform references: `terraform_remote_state` and provider-native data sources. Both can resolve the same values. The choice between them has security and architectural consequences that are not immediately obvious.

stratum-platform is a public repository. Both platform foundation repositories are also public. This is a deliberate portfolio decision — the work is publicly demonstrable evidence of engineering capability.

## Decision

All cross-repository references in stratum-platform use provider-native data sources exclusively:

- `azurerm_resource_group`, `azurerm_virtual_network`,`azurerm_subnet` and equivalent resources for Azure references
- `aws_vpc`, `aws_subnet`, `aws_iam_role` and equivalent resources for AWS references

`terraform_remote_state` is not used for any cross-repository reference in this programme. It is used within individual repositories for cross-module references where the consuming module is in the same repo and state access is controlled.

Example of the pattern in practice:

```hcl
# Consuming a platform VNet in stratum-platform — Azure
data "azurerm_virtual_network" "hub" {
  name                = "vnet-hub"
  resource_group_name = "rg-platform-connectivity"
}

# Consuming a platform VPC in stratum-platform — AWS
data "aws_vpc" "platform" {
  tags = {
    Name = "vpc-platform"
  }
}
```

No backend configuration, no state file access, no credentials scoped to remote state storage in the platform repositories.

## Alternatives Considered

**`terraform_remote_state`**
— reads outputs directly from another module's state file stored in Azure Blob Storage or S3. Rejected for public repositories for the following reason: a state file contains every attribute of every resource Terraform manages, including values marked `sensitive`. Resource IDs, connection strings, principal IDs, and subscription identifiers are all present in plaintext in a state file. Granting stratum-platform read access to azure-landing-zone's state file in a public repository exposes the complete infrastructure inventory of the platform to anyone who can read the repository configuration and access the storage account.

`terraform_remote_state` is appropriate within a single repository where state access is controlled and the consuming module is not public. It is not appropriate as a cross-repository mechanism in a public portfolio.

**Hardcoded resource IDs**
— embedding resource IDs, VNet names, and subnet CIDRs directly in workload Terraform configuration. Rejected. Hardcoded values break on any infrastructure change in the platform foundation, require manual updates across multiple files, and provide no validation that the referenced resource actually exists at plan time. Data sources fail loudly at plan if the referenced resource does not exist — hardcoded values fail silently at apply or produce incorrect
behaviour.

**Shared variable files**
— passing platform outputs through shared tfvars files consumed by stratum-platform. Rejected. This couples the deployment process of stratum-platform to the output of a manual step in the platform repositories. It also creates a stale reference problem: tfvars are not updated automatically when platform infrastructure changes.

## Consequences

**Positive:**

- No state file contents are exposed through stratum-platform configuration. The repository can be fully public without revealing platform infrastructure inventory.
- Data sources validate at plan time — if the referenced resource does not exist or the name has changed, the plan fails with a clear error rather than silently deploying against stale values.
- stratum-platform has no dependency on the storage account credentials or access policies of the platform foundation repositories. The consuming repository needs only the provider credentials to read Azure or AWS resources — nothing more.
- The pattern is consistent across both clouds. Azure and AWS data sources behave identically from a workflow perspective, which simplifies the unified pipeline built in Phase 3.

**Negative:**

- Data sources require that the referenced infrastructure exists and is named correctly at plan time. If azure-landing-zone or aws-landing-zone has not been applied, stratum-platform plans will fail on data source resolution. This is an explicit ordering dependency: platform foundations must be deployed before stratum-platform can be planned.
- Resource names must be stable and agreed between repositories. A rename in the platform foundation breaks the data source in stratum-platform. Mitigated by treating platform resource names as a contract — changes require coordination and are documented in the platform repository's ADR or runbook.
- Data sources expose only what the provider API returns for a given resource type. If a platform output is not available as a data source attribute, an alternative mechanism is required. In practice this is rare for the resource types used in this programme.
